    AWSのCloud Volumes ONTAPのネットワーク要件 GitHubで編集 ドキュメントの変更をリクエストする

    06/12/2019 貢献者

    Cloud Volumes ONTAPシステムが正しく動作できるようにAWSネットワークを設定します。

    Cloud Managerがアクセスを必要とするエンドポイントのリストをお探しですか？それらは現在、単一の場所に維持されています。 詳細はこちら

    Cloud Volumes ONTAPのための一般的なAWSネットワーク要件

    AWSでは以下の要件を満たす必要があります。

    Cloud Volumes ONTAPノードへのアウトバウンドインターネットアクセス

    Cloud Volumes ONTAPノードでNetApp AutoSupportにメッセージを送信するには、アウトバウンドインターネットアクセスが必要です。これにより、ストレージの状態を予防的に監視できます。

    Cloud Volumes ONTAPがAutoSupportメッセージを送信できるように、ルーティングおよびファイアウォールポリシーでは、以下のエンドポイントへのAWS HTTP / HTTPSトラフィックを許可する必要があります。

    • https://support.netapp.com/aods/asupmessage

    • https://support.netapp.com/asupprod/post/1.0/postAsup

      NATインスタンスがある場合は、プライベートサブネットからインターネットへのHTTPSトラフィックを許可するインバウンドセキュリティグループルールを定義する必要があります。

    HAメディエータのアウトバウンドインターネットアクセス

    HAメディエータインスタンスは、ストレージフェイルオーバーを支援できるように、AWS EC2サービスへのアウトバウンド接続が必要です。接続を提供するには、パブリックIPアドレスを追加するか、プロキシサーバーを指定するか、手動のオプションを使用します。

    手動オプションは、ターゲットサブネットからAWS EC2サービスへのNATゲートウェイまたはインターフェイスVPCエンドポイントです。 VPCエンドポイントの詳細については、 AWSドキュメント：インターフェイスVPCエンドポイント（AWS PrivateLink）を参照してください。

    セキュリティグループ

    Cloud Managerがそれを行うので、セキュリティグループを作成する必要はありません。独自のものを使用する必要がある場合は、 セキュリティグループの規則を参照してください。

    データ階層化のためのCloud Volumes ONTAPからAWS S3への接続

    EBSをパフォーマンス層として使用し、AWS S3をキャパシティ層として使用する場合は、Cloud Volumes ONTAPがS3に接続されていることを確認する必要があります。その接続を提供する最善の方法は、S3サービスへのVPCエンドポイントを作成することです。手順については、 AWSドキュメント：ゲートウェイエンドポイントの作成を参照してください。

    VPCエンドポイントを作成するときは、Cloud Volumes ONTAPインスタンスに対応するリージョン、VPC、およびルートテーブルを必ず選択してください。セキュリティグループを変更して、S3エンドポイントへのトラフィックを有効にするアウトバウンドHTTPSルールを追加する必要もあります。そうでないと、Cloud Volumes ONTAPはS3サービスに接続できません。

    問題が発生した場合は、 AWSサポートナレッジセンター：ゲートウェイVPCエンドポイントを使用してS3バケットに接続できないのはなぜですか。

    他のネットワークのONTAPシステムへの接続

    AWSのCloud Volumes ONTAPシステムと他のネットワークのONTAPシステムとの間でデータを複製するには、AWS VPCと他のネットワーク（Azure VNetや企業ネットワークなど）との間にVPN接続が必要です。手順については、 AWSのドキュメント：AWS VPN接続の設定を参照してください。

    CIFS用のDNSとActive Directory

    CIFSストレージをプロビジョニングする場合は、AWSにDNSとActive Directoryを設定するか、オンプレミス設定をAWSに拡張する必要があります。

    DNSサーバーはActive Directory環境に名前解決サービスを提供する必要があります。デフォルトのEC2 DNSサーバーを使用するようにDHCPオプションセットを構成できます。これは、Active Directory環境で使用されるDNSサーバーであってはなりません。

    手順については、AWSクラウドクイックスタートリファレンスデプロイのAWSドキュメント：Active Directoryドメインサービスを参照してください。

    複数のAZでのCloud Volumes ONTAP HAのAWSネットワーク要件

    追加のAWSネットワーキング要件は、複数のアベイラビリティーゾーン（AZ）を使用するCloud Volumes ONTAP HA構成に適用されます。 Cloud Managerでネットワークの詳細を入力する必要があるため、HAペアを起動する前にこれらの要件を確認する必要があります。

    HAペアの仕組みを理解するには、 高可用性ペアを参照してください。

    アベイラビリティーゾーン

    このHA配置モデルは、データの高可用性を確保するために複数のAZを使用します。 HAペア間の通信チャネルを提供する各Cloud Volumes ONTAPインスタンスとメディエータインスタンスには専用のAZを使用する必要があります。

    NASデータとクラスタ/ SVM管理用のフローティングIPアドレス

    複数のAZのHA構成は、障害が発生した場合にノード間を移動する浮動IPアドレスを使用します。 AWSトランジットゲートウェイを設定しない限り、これらはVPCの外部からネイティブにアクセスすることはできません。

    1つのフローティングIPアドレスはクラスタ管理用、1つはノード1のNFS / CIFSデータ用、もう1つはノード2のNFS / CIFSデータ用です。SVM管理用の4番目のフローティングIPアドレスはオプションです。

    HAペアでSnapDrive for WindowsまたはSnapCenterを使用する場合は、SVM管理LIFに浮動IPアドレスが必要です。システムを展開するときにIPアドレスを指定しない場合は、後でLIFを作成できます。詳細については、「 Cloud Volumes ONTAPのセットアップ 」を参照してください。

    Cloud Volumes ONTAP HAの作業環境を作成するときには、Cloud ManagerにフローティングIPアドレスを入力する必要があります。 Cloud Managerは、システムの起動時にHAペアにIPアドレスを割り当てます。

    フローティングIPアドレスは、HA設定をデプロイするAWSリージョン内のすべてのVPCのCIDRブロックの外側にある必要があります。フローティングIPアドレスは、地域のVPCの外側にある論理サブネットとして考えてください。

    次の例は、AWSリージョン内のフローティングIPアドレスとVPCの関係を示しています。フローティングIPアドレスはすべてのVPCのCIDRブロックの外側にありますが、ルートテーブルを介してサブネットにルーティングできます。

    Cloud Managerは、iSCSIアクセス用およびVPC外部のクライアントからのNASアクセス用の静的IPアドレスを自動的に作成します。あなたはこれらのタイプのIPアドレスのためのどんな要件も満たす必要はありません。
    VPC外部からのフローティングIPアクセスを可能にするトランジットゲートウェイ

    HAペアが存在するVPCの外部からHAペアのフローティングIPアドレスへのアクセスを有効にするようにAWS中継ゲートウェイを設定します。

    ルートテーブル

    Cloud ManagerでフローティングIPアドレスを指定したら、フローティングIPアドレスへのルートを含める必要があるルートテーブルを選択する必要があります。これにより、HAペアへのクライアントアクセスが可能になります。

    VPC内のサブネットに対して1つのルートテーブル（メインルートテーブル）しかない場合、Cloud Managerは自動的にフローティングIPアドレスをそのルートテーブルに追加します。複数のルートテーブルがある場合は、HAペアを起動するときに正しいルートテーブルを選択することが非常に重要です。そうでないと、一部のクライアントがCloud Volumes ONTAPにアクセスできない可能性があります。

    たとえば、異なるルートテーブルに関連付けられている2つのサブネットがあるとします。ルートテーブルBを選択せずにルートテーブルAを選択した場合、ルートテーブルAに関連付けられているサブネット内のクライアントはHAペアにアクセスできますが、ルートテーブルBに関連付けられているサブネット内のクライアントはアクセスできません。

    ルートテーブルの詳細については、 AWSドキュメント：Route Tablesを参照してください。

    NetApp管理ツールへの接続

    複数のAZにあるHA構成でNetApp管理ツールを使用するには、2つの接続オプションがあります。

    1. NetApp管理ツールを別のVPCにデプロイし、AWSトランジットゲートウェイを設定します 。ゲートウェイを使用すると、VPCの外部からクラスタ管理インターフェイスの浮動IPアドレスにアクセスできます。

    2. NASクライアントと同様のルーティング構成を使用して、NetApp管理ツールを同じVPCに展開します。

    サンプルVPC構成

    Cloud ManagerとCloud Volumes ONTAPをAWSにデプロイする方法をよりよく理解するために、最も一般的なVPC設定を確認する必要があります。

    • パブリックおよびプライベートサブネットとNATデバイスを備えたVPC

    • プライベートサブネットとネットワークへのVPN接続を備えたVPC

    パブリックおよびプライベートサブネットとNATデバイスを備えたVPC

    このVPC構成には、パブリックサブネットとプライベートサブネット、VPCをインターネットに接続するインターネットゲートウェイ、およびプライベートサブネットからのアウトバウンドインターネットトラフィックを可能にするパブリックサブネット内のNATゲートウェイまたはNATインスタンスが含まれます。この構成では、Cloud Managerをパブリックサブネットまたはプライベートサブネットで実行できますが、パブリックサブネットはVPC外のホストからのアクセスを許可するため推奨されています。その後、プライベートサブネットでCloud Volumes ONTAPインスタンスを起動できます。

    NATデバイスの代わりに、インターネット接続を提供するためにHTTPプロキシを使用することができます。

    このシナリオの詳細については、 AWSドキュメント：シナリオ2：パブリックサブネットとプライベートサブネット（NAT）を使用したVPCを参照してください。

    次の図は、パブリックサブネットで実行されているCloud Managerと、プライベートサブネットで実行されている単一ノードシステムを示しています。

    プライベートサブネットとネットワークへのVPN接続を備えたVPC

    このVPC構成は、Cloud Volumes ONTAPがプライベート環境の拡張となるハイブリッドクラウド構成です。構成には、プライベートサブネットと、ネットワークへのVPN接続を備えた仮想プライベートゲートウェイが含まれます。 VPNトンネルを介したルーティングにより、EC2インスタンスはネットワークおよびファイアウォールを介してインターネットにアクセスできます。プライベートサブネットまたはデータセンターでCloud Managerを実行できます。その後、プライベートサブネットでCloud Volumes ONTAPを起動します。

    この構成でプロキシサーバーを使用してインターネットアクセスを許可することもできます。プロキシサーバーはデータセンターまたはAWSにあります。

    データセンターのFASシステムとAWSのCloud Volumes ONTAPシステムの間でデータを複製する場合は、リンクが安全になるようにVPN接続を使用する必要があります。

    このシナリオの詳細については、 AWSドキュメンテーション：シナリオ4：プライベートサブネットのみを使用するVPCとAWS管理VPNアクセスを参照してください 。
