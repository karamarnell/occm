    データ階層化の概要 GitHubで編集 ドキュメントの変更をリクエストする

    06/18/2019 貢献者

    非アクティブなデータを低コストのオブジェクトストレージに自動階層化できるようにすることで、ストレージコストを削減します。アクティブデータは高性能SSDまたはHDDに残りますが、非アクティブデータは低コストのオブジェクトストレージと連携します。これにより、1次ストレージのスペースを再利用し、2次ストレージを縮小することができます。

    クラウドボリュームONTAPは、AWS、Azure、およびGoogle Cloud Platformのデータ階層化をサポートしています。データ階層化は、FabricPoolテクノロジによって強化されています。

    データ階層化を有効にするために機能ライセンスをインストールする必要はありません（FabricPool）。

    AWSのデータ階層化

    AWSでデータ階層化を有効にすると、Cloud Volumes ONTAPはホットデータのパフォーマンス階層としてEBSを使用し、非アクティブデータの容量階層としてAWS S3を使用します。システムの階層化レベルを変更すると、別のS3ストレージクラスを選択できます。

    パフォーマンス層

    パフォーマンス層は、汎用SSD、プロビジョンドIOPS SSD、またはスループット最適化HDDです。

    キャパシティティア

    Cloud Volumes ONTAPシステムは、 標準ストレージクラスを使用して、非アクティブデータを単一のS3バケットに階層化します。標準は、複数のアベイラビリティーゾーンにまたがって格納されている頻繁にアクセスされるデータに最適です。

    Cloud Managerは、作業環境ごとに単一のS3バケットを作成し、それにfabric-pool- cluster固有の識別子を付けます 。ボリュームごとに異なるS3バケットは作成されません。
    階層化レベル

    非アクティブデータへのアクセスを計画していない場合は、システムの階層化レベルをインテリジェント階層化、 1ゾーン低頻度アクセス 、または標準低頻度アクセスのいずれかに変更することで、ストレージコストを削減できます。階層化レベルを変更すると、非アクティブデータは標準ストレージクラスから開始され、30日後にデータにアクセスしない場合は選択したストレージクラスに移動します。

    データにアクセスするとアクセスコストが高くなるため、階層化レベルを変更する前にそれを考慮してください。 Amazon S3ストレージクラスの詳細を学んでください 。

    システムを作成した後に階層化レベルを変更することは可能です。詳しくは、 非アクティブ・データを低コストのオブジェクト・ストレージに階層化するを参照してください。

    階層化レベルはシステム全体であり、ボリューム単位ではありません。

    Azureのデータ階層化

    Azureでデータ階層化を有効にすると、Cloud Volumes ONTAPは、Azure管理対象ディスクをホットデータのパフォーマンス階層として、およびAzure Blobストレージを非アクティブデータの容量階層として使用します。システムの階層化レベルを変更すると、別のAzureストレージ階層を選択できます。

    パフォーマンス層

    パフォーマンス層は、SSDまたはHDDのいずれかです。

    キャパシティティア

    Cloud Volumes ONTAPシステムは、Azure ホットストレージ層を使用して、非アクティブデータを単一のBlobコンテナーに階層化します。ホットティアは頻繁にアクセスされるデータに最適です。

    Cloud Managerは、各Cloud Volumes ONTAP作業環境用に単一のコンテナーを持つ新しいストレージアカウントを作成します。ストレージアカウントの名前はランダムです。ボリュームごとに異なるコンテナーは作成されません。
    階層化レベル

    非アクティブなデータにアクセスする予定がない場合は、システムの階層化レベルをAzure クールストレージ層に変更することで、ストレージコストを削減できます。階層化レベルを変更すると、非アクティブデータはホットストレージ層から開始され、30日経ってもデータにアクセスしない場合はクールストレージ層に移動します。

    データにアクセスするとアクセスコストが高くなるため、階層化レベルを変更する前にそれを考慮してください。 Azure Blobストレージアクセス層の詳細を学んでください 。

    システムを作成した後に階層化レベルを変更することは可能です。詳しくは、 非アクティブ・データを低コストのオブジェクト・ストレージに階層化するを参照してください。

    階層化レベルはシステム全体であり、ボリューム単位ではありません。

    GCPでのデータ階層化

    GCPでデータ階層化を有効にすると、Cloud Volumes ONTAPはホットディスクのパフォーマンス階層として永続ディスクを使用し、非アクティブデータの容量階層としてGoogle Cloud Storageバケットを使用します。

    パフォーマンス層

    パフォーマンス層は、SSDまたはHDD（標準ディスク）のいずれかです。

    キャパシティティア

    Cloud Volumes ONTAPシステムは、 地域ストレージクラスを使用して、非アクティブデータを単一のGoogle Cloud Storageバケットに階層化します。

    Cloud Managerは、作業環境ごとに1つのバケットを作成し、それにファブリックプールクラスター固有の識別子を付けます 。ボリュームごとに異なるバケットは作成されません。
    階層化レベル

    現時点では、他のGCPストレージクラスはサポートされていません。

    データ階層化と容量制限

    データ階層化を有効にしても、システムの容量制限は変わりません。制限はパフォーマンス層と容量層に分散しています。

    ボリューム階層化ポリシー

    データ階層化を有効にするには、ボリュームを作成、変更、または複製するときにボリューム階層化ポリシーを選択する必要があります。ボリュームごとに異なるポリシーを選択できます。

    一部の階層化ポリシーには、関連付けられた最小冷却期間があります。この期間は、データが「コールド」と見なされ、容量階層に移動されるためにボリューム内のユーザーデータが非アクティブのままでいる時間を設定します。

    Cloud Managerでは、ボリュームを作成または変更するときに、次のボリューム階層化ポリシーから選択できます。

    スナップショットのみ

    アグリゲートの容量が50％に達すると、Cloud Volumes ONTAPはアクティブファイルシステムに関連付けられていないSnapshotコピーのコールドユーザーデータを容量階層に階層化します。冷却期間は約2日です。

    読み込まれると、キャパシティ層のコールドデータブロックがホットになり、パフォーマンス層に移動されます。

    自動

    アグリゲートの容量が50％に達すると、Cloud Volumes ONTAPはボリューム内のコールドデータブロックを容量階層に階層化します。コールドデータには、Snapshotコピーだけでなく、アクティブファイルシステムからのコールドユーザーデータも含まれます。冷却期間は約31日です。

    このポリシーは、Cloud Volumes ONTAP 9.4以降でサポートされています。

    ランダム読み取りで読み取ると、キャパシティ層のコールドデータブロックがホットになり、パフォーマンス層に移動します。インデックススキャンやウイルス対策スキャンに関連するものなど、シーケンシャルリードで読み取ると、コールドデータブロックはコールドのままになり、パフォーマンス層に移動しません。

    無し

    パフォーマンス層のボリュームのデータを保持し、容量層に移動されないようにします。

    ボリュームを複製するときに、DRデータをオブジェクトストレージに階層化するかどうかを選択できます。変更した場合、Cloud Managerはバックアップポリシーをデータ保護ボリュームに適用します。 Cloud Volumes ONTAP 9.6以降では、 All階層化ポリシーがバックアップポリシーに置き換わります。

    データ階層化の設定

    手順およびサポートされている構成のリストについては、 非アクティブデータを低コストのオブジェクトストレージに階層化するを参照してください。
